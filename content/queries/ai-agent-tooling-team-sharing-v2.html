01
把个人品味变成 AI 的全局规则
一套 AGENTS.md，管所有 AI 工具
😤 痛点
  • 每次开新会话，都要重新告诉 AI "别废话"、"结论先行"、"代码用英文"
  • Hermes 和 Claude Code 回答风格完全不一样，像跟两个人说话
  • AI 太客气——你说错了它还在 "Great question!"
  • 你希望 AI 帮你守住标准，但它根本不记得上次说过什么
💡 核心问题

每个 AI 工具都是独立的，它们不共享你的偏好、你的原则、你的习惯。

🔑 灵感：Hermes 的 SOUL.md

Hermes Agent 有个 SOUL.md 文件，定义 Agent 的"灵魂"：角色、原则、风格、边界。注入到系统提示词第一个 slot，每次对话都生效。

但 SOUL.md 只作用于 Hermes。 用 Claude Code、OpenCode 时这些规则就失效了。

✅ 我的做法：全局 AGENTS.md

把 SOUL.md 的思路抽出来，写成 ~/.claude/CLAUDE.md——Claude Code 和 OpenCode 都自动加载。一份文件，多个工具引用，同一套行为规范。

📋 九大板块

板块说什么强制程度
§1 角色严谨的思考伙伴，不是聊天机器人所有工具
§2 核心原则7 条行为准则所有工具
§3 输出风格结论先行、零废话、5 秒可扫描所有工具
§4 反模式干预主动推回"囤积资源"、"重组代替使用"所有工具
§5 用户关怀效率优先、疲惫时建议轻任务所有工具
§6 已知矛盾5 对真实矛盾，观察不强行解决所有工具
§7 个人档案渐进式构建用户画像渐进增强
§8 技能提取从交互中提炼 Skill渐进增强
§9 知识复利操作知识库前读约束文件渐进增强

⭐ 七条核心原则

1. 先对齐再行动

复杂任务先确认理解再动手

2. 信息时效优先

优先最近 3 个月官方源

3. 复利而非堆积

每次交互让知识库更好

4. 精确优先于礼貌

直接说"这是错的，因为 X"

5. 验证行动优于空转

用输出验证，不是用输入验证

6. 深度优先于广度

精通一个 > 浅尝十个

7. 外科手术式修改

只动必须动的，匹配现有风格，用户不会注意到的改动就跳过

⚠️ 关键设计：分层适用

§1-6 是硬规则，所有工具必须遵守。§7-9 涉及跨会话积累——Claude Code 没有跨会话记忆，如果硬规则要求它做，就会报错或造假。

所以 §7-9 设计成软规则：持久化 Agent（Hermes）自动执行，会话型工具（Claude Code、OpenCode）仅建议，由用户确认。

🎯 一句话总结

与其每次跟 AI 重复约定规则，不如把"品味"编码成全局文件。SOUL.md 管 Hermes，AGENTS.md 管所有工具——同一套标准，分层适配。

02
让 AI 替我跑腿
Hermes Gateway + 定时任务 = 自动化资讯采集
😤 痛点
  • 信息源太分散：HN、arXiv、GitHub Trending、技术博客……每天刷 1-2 小时
  • 刷了不等于看了：收藏一堆，消化不到 20%——"囤积不消化"
  • 碎片时间没利用：通勤时可以做信息筛选，但工具不趁手

🏗️ Hermes Gateway 架构

Gateway 是常驻后台进程，同时连接 20+ 通讯平台。我主要用 Telegram 和飞书。

Hermes Gateway（后台进程）
↙ ↙ ↙
Telegram Bot + 飞书 Bot + Cron 调度器

🤖 创建 Bot

📱 Telegram Bot

1. @BotFather → /newbot
2. hermes gateway setup
3. hermes gateway start

💬 飞书 Bot

1. open.feishu.cn 创建应用
2. 配置权限 + 订阅事件
3. hermes gateway setup

💡 关键细节

飞书用 websocket 模式连接，不需要公网 IP，不需要回调地址——内网环境友好。

⏰ 定时任务：Cron 调度器

Gateway 每 60 秒 tick 一次，到期任务在隔离 Agent 会话中执行，结果投递到指定平台。

每天早上 9 点，搜集 AI/ML 热点
hermes cron create "0 9 * * *" \
  "搜索 Hacker News 和 arXiv 上最近 24 小时的 AI/ML 热点，总结 top 5，每条附上来源链接" \
  --deliver feishu \
  --name "Daily AI News"
每 6 小时，检查技术博客更新
hermes cron create "0 */6 * * *" \
  "检查 Lilian Weng、Karpathy、Simon Willison 的博客是否有新文章，有则总结要点" \
  --deliver telegram \
  --name "Blog Watcher"

直接在聊天里说 "Every morning at 9am, check HN for AI news and send me a summary on Telegram"，Hermes 自动创建 Cron 任务。

🔄 闭环：采集 → 审核 → 归档

Cron 定时采集 → Agent 总结筛选
↓
投递到 Telegram / 飞书
↓
我审核后确认有价值
↓
"归档这篇到知识库" → Agent 创建/更新 Wiki 页面

人机分工：Agent 初筛（总结 + 标注重要性），我终审。人类策展，Agent 执行。

高级用法

  • 任务链：Job A 输出 → Job B 输入（context_from）
  • Skill 绑定：--skill blogwatcher，按最佳实践执行
  • 无 Agent 模式：no_agent=True，纯脚本，零 LLM 成本
  • 静默模式：返回 [SILENT] 时不投递（"有变化才通知"）
🎯 一句话总结

Hermes Gateway 把 AI 从"你问我答"变成"主动服务"——Bot 随时触达，Cron 替你跑腿。信息搜集从"每天刷 2 小时"变成"AI 替我筛，我只看精华"。

03
给 AI 装一套开发方法论
SuperPowers + 4 个设计 Skill = AI 开发纪律
😤 痛点
  • "帮我做登录页面" → AI 二话不说就写代码，没问用户群体、设计语言
  • 改 bug 试了三种方案不行，第四种"应该能过"——还是不行
  • "应该没问题" → 一跑满屏报错
  • 复杂功能一口气 500 行，根本没法 review
💡 核心问题

AI 写代码太"急"——跳过思考、设计、测试、验证，直接产出代码。不是效率高，是隐患大。

🦸 SuperPowers：开发方法论，不是 Skill 合集

来自 obra/superpowers，本质上是一套给 AI 编码 Agent 用的软件开发方法论。三个核心机制：

🔌 机制一：Bootstrap 注入

每次会话开始自动注入方法论到 Agent 上下文。规则不靠重复约定，靠一次编码、持续生效。

🚧 机制二：强制流水线

每个阶段有硬门禁（HARD GATE），设计没通过不许写代码，测试没写不许写产品代码。

用户需求
↓
brainstorming（设计）
🚧 硬门禁：设计没通过，不许写代码
↓
writing-plans（拆任务，2-5 分钟粒度）
↓
TDD + 实现（先写测试，再写代码）
↓
code-review + verification（审查 + 验证）
🧠 机制三：反合理化设计

AI 会给自己找理由跳过纪律。每个 Skill 都有铁律 + 反合理化表 + 红旗信号 + 禁止用语列表——对抗性提示工程，用 AI 自己的推理能力约束自己。

AI 内心的想法现实
"这只是个简单问题"问题也是任务，检查有没有适用的 Skill
"这个 Skill 太大题小做了"简单的事情会变复杂，用它
"我先了解一下上下文"Skill 会告诉你怎么了解，先检查
"我记得这个 Skill 的内容"Skill 会更新，读最新版本

类别Skill作用
元技能using-superpowersBootstrap 注入，"先检查再行动"

🎨 4 个设计领域 Skill

🔍 find-skills

搜索社区生态，找到需要的 Skill

从交互中提炼新 Skill，经验不丢失

production-grade 前端代码，拒绝 AI 通用审美

+

+

🎯 一句话总结

SuperPowers 是一套开发方法论——强制流水线、硬门禁、反合理化设计，让 AI 从"拿到需求就写代码"变成"先设计、再计划、先测试、再实现、必须验证"。

04
让 AI 越来越懂你
Karpathy LLM Wiki + 个人画像 + Skill 提炼 = 知识复利
😤 痛点
  • 上周跟 AI 讨论的方案，这周再问它完全不记得
  • AI 不了解你的背景，每次从头解释"我用 Java 12 年了"
  • 纠正了 AI 的工作流程，下次会话它又按老方法来
  • 积累很多对话，但都是碎片，搜不到、用不上

📖 Karpathy 的 LLM Wiki 理念

维度RAGWiki
知识发现每次查询从零检索编译一次，持续更新
交叉引用缺失，碎片化已建好，矛盾已标记
增长模式线性堆积复利增长
可审查性黑盒人可以浏览、验证、修正
📐 复利公式

知识库价值 = 页面数量 × 交叉引用密度 × 内容质量

transcripts/ papers/ articles/

🧩 三个"越来越懂你"的机制

👤 机制一：个人画像

profile/ 目录：identity、preferences、tech-stack、goals、interests。AI 知道你是 Java 12 年老兵，不会解释面向对象。

⚡ 机制二：Skill 提炼

隐性经验（调试路径、方案取舍）沉淀成可执行的 Skill。提炼 → 审核 → 调用 → 迭代 → 整合。

📝 机制三：会话沉淀（双轨制）

Wiki 页面（显性知识：技术调研、方案对比）+ Skill 提炼（隐性经验：调试路径、工作流程）。新页面至少 2 个出站链接——织网不堆文件。

🔍 Lint 健康检查（11 项）

结构健康

孤儿页面 · 断链 · 索引完整性 · 页面大小 · 日志轮换

内容质量

Frontmatter 验证 · 标签蔓延 · 陈旧内容 · 矛盾审查 · 质量信号 · Source drift

🎯 一句话总结

Wiki 的核心是复利——每次交互让知识网络更密。个人画像让 AI 了解你是谁，Skill 提炼让 AI 记住你怎么做事，会话沉淀让知识不随会话消失。

OpenCode 做内容准备，Google AI Pro 做富媒体输出
😤 痛点
  • 技术方案写好 Markdown，但汇报需要演示文稿
  • 分享文稿如果能生成音频，通勤时自己也能听一遍

Google AI Pro 年费会员（无 API Key 额度）。策略：能集成的全自动，集成不了的半自动——OpenCode 生成提示词，手动导入 Google Web UI。

06
架构师的 AI 武器库
4+1 视图 + Diagram-as-Code = 可编辑、可 diff、可版本控制
😤 痛点
  • Visio 画的图不可 diff——版本管理靠文件名后缀 _v2_final_真的final.pptx
  • AI 生成的图片不可编辑——迭代 10 轮，每轮重新生成
  • 4+1 视图工作量大——5 张图画完半天没了
💡 核心原则

Agent 输出代码，不输出图片。代码永远可编辑、可 diff、可版本控制。图片不行。
