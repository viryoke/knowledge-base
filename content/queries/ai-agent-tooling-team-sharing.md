## 开场：为什么要聊这个？

先说背景。我日常的工作流里，AI 已经不是"偶尔问一下"的角色了，而是**常驻的工作伙伴**。从写代码、查资料、做架构设计到整理学习笔记，几乎每个环节都有 AI Agent 参与。

但问题来了——**工具太多了**。Claude Code、Copilot、Cursor、各种 ChatBot……每个都能干活，但怎么组合使用效率最高？怎么让 AI 不只是"一次性回答机器"，而是真正积累起来？

这就是我这段时间在摸索的事。今天分享的五个部分：

1. **工具怎么选** — 我用的三个工具和各自适合的场景
2. **Skill 生态** — 让 AI 能力可复用的关键
3. **知识库建设** — 为什么我选了 Wiki 而不是 RAG
4. **全局约束** — 一套规则管所有工具
5. **九个具体问题** — 从需求理解到实战输出的完整踩坑链

## Part 1：工具选型——不是选最好的，是选最配的

### 我在用的三个工具

| 工具 | 一句话定位 | 我主要拿它干嘛 |
|------|-----------|---------------|
| **Hermes Agent** | 持久化 Agent，随时在线 | 碎片化问答、多端协作（Telegram/飞书都能用）、跨会话记忆 |
| **Claude Code** | 会话型深度编码 | 复杂重构、大规模代码分析、需要长上下文的硬核任务 |
| **OpenCode** | 可扩展工作台 | Skill 编排、多模型切换、自定义工作流 |

### 关键认知：三种模式，不是三个竞品

这三个工具**不是替代关系**，对应的是三种工作模式：

- **持久陪伴模式**（Hermes）→ 碎片时间、随时触达、跨会话延续
- **深度攻坚模式**（Claude Code）→ 坐下来一次性啃硬骨头
- **可扩展工作台模式**（OpenCode）→ 需要灵活编排、自定义流程

### 我的日常协作模式

```
碎片问题 → 打开 Hermes（手机上 Telegram 直接问）
    ↓ 发现这个问题比较复杂
深度分析 → 切到 Claude Code（坐下来认真搞）
    ↓ 搞完了发现这个流程可以复用
沉淀流程 → OpenCode（写成 Skill）
    ↓ 经验写进知识库
所有工具共享 → AGENTS.md 统一约束
```

**核心观点：别纠结"哪个工具最好"，想清楚"我现在处于哪种模式"。**

## Part 2：Skill 生态——让 AI 越用越聪明的关键

### Skill 是什么？

简单说：**Skill 就是把你的最佳实践写成 AI 能直接加载执行的代码**。

没有 Skill 的世界：每次遇到类似问题，都要重新跟 AI 解释背景、约定格式、纠正错误。
有 Skill 的世界：AI 自动识别场景 → 加载对应的 Skill → 直接按最佳实践执行。

### 我用的 5 个核心 Skill

| Skill | 干什么的 | 解决了什么痛点 |
|-------|---------|---------------|
| **SuperPowers** | 自动识别该用哪个 Skill | 不用记住所有 Skill 名字，AI 自己匹配 |
| **find-skills** | 搜索社区有什么可用的 Skill | 已有 Skill 不够用时，去社区找 |
| **Skill-Creator** | 从交互中提炼新 Skill | 经验不丢失，一个人的探索全队受益 |
| **frontend-design** | 生成高质量前端代码 | 避免 AI 写出来的页面都是"一个样" |
| **ui-ux-pro-max** | 50+ 设计风格 + 色彩方案 + 字体搭配 | 让 AI 的设计输出有专业依据 |

### 这 5 个 Skill 形成了一条能力链

```
发现 → 加载 → 提炼 → 执行 → 反馈迭代
find-skills  SuperPowers  Skill-Creator  领域Skill  回到Skill-Creator
```

**核心观点：Skill 的本质是"把最佳实践编码化"。单个 Skill 有用，但能力链（发现→加载→提炼→执行）才是真正让 AI 越用越强的飞轮。**

## Part 3：知识库——为什么我选了 Wiki 而不是 RAG

### 先说结论

我用的是 Karpathy 提出的 LLM Wiki 方案，不用 RAG。原因很简单：

| 维度 | RAG | Wiki |
|------|-----|------|
| 知识发现 | 每次查询从零检索 | 编译一次，持续更新 |
| 交叉引用 | 碎片化 | 已建好，矛盾已标记 |
| 增长模式 | 线性堆积 | **复利增长** |
| 可审查性 | 黑盒 | 人可以浏览、验证、修正 |

### 复利公式

**知识库价值 = 页面数量 × 交叉引用密度 × 内容质量**

每次新增一个页面，不只是多了一篇文档，而是更新了所有相关页面的链接。知识网络越织越密，而不是文档越堆越多。

### 三层架构

```
Layer 3 — Schema（约束文件：SCHEMA.md + AGENTS.md）
    ↓ 约束 Agent 的行为规范
Layer 2 — Wiki（加工后的知识页面）
    ↑ 只读引用
Layer 1 — Raw Sources（原始素材，不可变）
```

### 当前规模

- **106 页**，覆盖 AI/ML、软件工程、英语、备考等领域
- 20+ 课程页面，60+ 概念页面
- 每个页面至少 2 个出站链接，形成知识网络

**核心观点：Wiki 优于 RAG 的核心是"复利"——每次交互让知识网络更密，而不是文档更多。RAG 是"用多少查多少"，Wiki 是"用一次赚一次"。**

## Part 4：全局约束——一套 AGENTS.md 管所有工具

### 灵感来源

Hermes Agent 有个 SOUL.md 文件，定义 Agent 的"灵魂"——角色、原则、边界。我借鉴了这个思路，写了一套 **AGENTS.md** 作为全局约束，所有 AI 工具共享。

### 九大板块速览

| 板块 | 说什么 | 强制程度 |
|------|--------|---------|
| §1 角色 | 严谨的思考伙伴，不是聊天机器人 | 所有工具 |
| §2 核心原则 | 7 条行为准则 | 所有工具 |
| §3 输出风格 | 结论先行、零废话、5 秒可扫描 | 所有工具 |
| §4 反模式干预 | 主动推回"囤积"、"重组代替使用"等坏习惯 | 所有工具 |
| §5 用户关怀 | 效率优先、疲惫时建议轻任务 | 所有工具 |
| §6 已知矛盾 | 5 对真实矛盾，观察不强行解决 | 所有工具 |
| §7 个人档案 | 渐进式构建用户画像 | 渐进增强 |
| §8 技能提取 | 从交互中提炼 Skill | 渐进增强 |
| §9 知识复利 | 操作知识库前读约束文件 | 渐进增强 |

### 七条核心原则（我觉得最值得分享的）

1. **先对齐再行动** — 复杂任务先确认理解再动手
2. **信息时效优先** — 优先最近 3 个月官方源
3. **复利而非堆积** — 每次交互让知识库更好
4. **精确优先于礼貌** — 直接说"这是错的"，不知道就说不知道
5. **验证行动优于空转** — 用输出验证，不是用输入验证
6. **深度优先于广度** — 精通一个 > 浅尝十个
7. **外科手术式修改** — 只动必须动的

### 分层适用设计

这是个关键设计：
- **§1-6 是硬规则**：所有工具必须遵守
- **§7-9 是软规则**：持久化 Agent 自动执行，会话型工具仅建议

为什么这么分？因为 Claude Code 没法自动更新你的个人档案（没有跨会话记忆），但 Hermes 可以。如果硬规则要求所有工具都做跨会话积累，Claude Code 就会报错或造假。**分层设计让约束适配工具能力，而不是反过来。**

## Part 5：九个核心问题——一条完整的踩坑链

这 9 个问题按**信息生命周期**排序，每个问题承上启下：

### ① 意图校准
**痛点**：AI 听到关键词就开始干活，跑偏了才发现。
**解法**："先对齐再行动"原则。复杂任务先列计划确认，简单任务不矫枉过正。

### ② Web Search 能力接入
**痛点**：AI 只能抓已知 URL，不能主动搜索发现。
**解法**：接入搜索 API。我的选择——

| 场景 | 推荐 |
|------|------|
| 日常搜索（90%） | Tavily（AI 原生，一行命令接入） |
| 技术深度查询 | Exa（语义搜索，技术文档质量最高） |
| 高频/省钱 | SearXNG 自托管（无限免费） |

### ③ 知识时效性
**痛点**：LLM 训练数据有截止日期，技术信息衰减极快。
**解法**：约束 AI "先搜索，再回答"，优先 3 个月内官方源，必须标注来源和日期。

### ④ 文档处理
**痛点**：素材格式五花八门（PDF、HTML、字幕）。
**解法**：文本优先原则 + 统一处理流水线。所有素材最终转 Markdown。
**关键约束**：`raw/` 目录只读不改，修正写进 Wiki 页面。

### ⑤ 知识沉淀
**痛点**：会话结束知识就丢了。
**解法**：双轨沉淀——Wiki 页面（显性知识）+ Skill 提炼（隐性经验）。
**复利关键**：每个新页面至少 2 个出站链接，织网不堆文件。

### ⑥ 经验积累
**痛点**：调试经验、架构决策过程随会话丢失。
**解法**：Skill 生命周期闭环（提炼→审核→调用→迭代→整合）。
**提炼信号**：用户纠正了 AI 的流程 / 出现了非平凡方案 / 复杂任务完成。

### ⑦ 绘图能力接入
**痛点**：AI 默认纯文本输出，但汇报和评审需要视觉输出。
**解法**：

| 场景 | 推荐 |
|------|------|
| Agent 原生集成 | FLUX MCP Server |
| 技术图理解力优先 | OpenAI gpt-image-2 |
| 高频/省钱 | Stable Diffusion 自托管 |

### ⑧ 结构图可编辑方案
**痛点**：AI 生成的图片不可编辑、不可 diff。
**解法**：diagram-as-code——

| 场景 | 方案 |
|------|------|
| 快速图 / 到处可渲染 | Mermaid（LLM 生成最可靠） |
| 复杂架构 / 4+1 视图 | D2（嵌套容器表达力强） |
| 需要手动微调 | Excalidraw（唯一支持渲染后编辑） |

**重要发现**：Structurizr（C4 模型官方工具）已于 2026.3 归档。

**核心原则：Agent 输出代码，不输出图片。代码永远可编辑、可 diff、可版本控制。**

### ⑨ 使用场景实战

三个典型场景串联以上所有能力：

**场景一：解决方案设计**
意图校准 → 信息检索 → 方案对比 → 结构化输出 → 知识沉淀

**场景二：4+1 视图绘制**
Agent 生成 5 个视图的代码（逻辑/开发/进程/物理/场景）→ 人审核 → 渲染

**场景三：结果可视化**
Skill 生成 HTML → 内嵌 Mermaid 图表 → 部署 GitHub Pages → 跨终端访问
